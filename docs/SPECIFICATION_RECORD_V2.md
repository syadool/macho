# MACHO - 記録画面リニューアル (V2) 仕様書

> 親仕様書: [SPECIFICATION.md](./SPECIFICATION.md)
> 本書は「4.3 記録画面」の詳細仕様。要件再定義（レスポンシブ・操作性の改善、セット単位の重量変更対応）に基づく記録画面の作り直しを扱う。

## 1. 背景

iPhone想定のモバイルファースト設計だが、旧UIは以下の点で操作性に不満があった。

- 重量・回数・セット数がエクササイズ単位の一括入力で、**セットごとに重量を変える**トレーニング（例: ピラミッド法、ドロップセット）に対応できなかった
- サブカテゴリ・器具の選択が入力の手間を増やしていた割に、使用頻度が低かった
- 有酸素運動の距離・カロリー入力は、体感的な負荷（時間）ほど重視されていなかった
- 種目名は毎回手入力で、過去の入力からのサジェストがなかった

好都合な事実として、DB（`workout_sets`）と保存バリデーション（`src/lib/workout-input.ts` の `normalizeStrengthSets`）は**既にセット単位の重量・回数を受け付ける設計**になっている（`NewExercisePayload.workout_sets: NewWorkoutSetPayload[]`）。そのため今回の変更はUI層が中心で、DBマイグレーションは不要。

## 2. 画面構成とUIフロー

記録は1画面（`app/record/record-form.tsx`）で完結する。フローは以下の順。

1. **日付選択**
   - 今日をデフォルト表示。既存の日付ピッカーを維持
   - 未来日は保存できない（既存バリデーション維持）

2. **筋トレ / 有酸素 タブ**
   - 既存のタブ切り替えUIを維持

3. **種目名入力（履歴サジェスト）**
   - テキスト入力フィールド。ハードコードされた初期値（例: 「インクラインベンチプレス」）は廃止し、空欄スタートにする
   - 入力欄フォーカス時 or 入力中に、過去の `workout_exercises` から取得した候補をドロップダウン表示
   - 候補をタップすると、種目名・部位（筋トレ時）・**前回のセット内容（重量・回数の行）**が自動的にフォームへセットされる
   - テンプレート機能（`app/templates/`）からの読み込みも従来通り併用可能。読み込んだテンプレートはper-set形式に変換してセット行リストへ反映する

4. **部位選択**（筋トレ時のみ）
   - 既存の3列グリッド（漢字アイコン + 英語ラベル、選択時ライムグリーンボーダー）を維持
   - **サブカテゴリのピル選択・器具のピル選択は廃止**（UIから削除。DBの `muscle_sub_groups` / `equipment` マスタ、`workout_exercises.muscle_sub_group_id` / `equipment_id` 列は温存し、保存時はnullを送る）

5. **セット行リスト（新設計、筋トレ時のみ）**
   - 各行 = `セット番号 | 重量(kg)ステッパー | 回数ステッパー`
   - 初期状態は1行（セット1）
   - 「+ セット追加」ボタンで行を追加。新しい行の初期値は**直前セットの値をコピー**する
   - 各行に削除ボタン（✕ もしくはスワイプ）を用意。ただし**最低1行は残す**（全削除は不可）
   - ステッパーのタップターゲットは44px相当に拡大し、数値部分をタップすると直接編集できるようにする

6. **有酸素入力**（有酸素時のみ）
   - 種目名 + **時間（分）のみ**
   - 距離・カロリーの入力欄は廃止（DBの `distance_km` / `calories` 列は温存。保存時は `distance_km: null, calories: null` を送る）

7. **追加済みエクササイズリスト**
   - 部位カラーバー + 種目名などの詳細を表示
   - セットごとの重量が異なる場合は「60kg×10, 65kg×8, 70kg×6」のようにセットごとの内訳を表示する
   - セットの重量が全て同じ場合は従来通り「60kg×10 ×3セット」のような簡潔表記でよい

8. **アクションボタン**
   - 「エクササイズを追加」（アウトラインボタン、追加済みリストへ積む）
   - 「ワークアウトを保存」（プライマリボタン、全エクササイズをまとめて保存）

## 3. データ仕様

- 保存時は `NewExercisePayload.workout_sets`（`NewWorkoutSetPayload[]`、要素は `{ weight_kg, reps }`）を使用する
- サーバー側 `src/lib/workout-input.ts` の `normalizeStrengthSets` は既に `exercise.workout_sets` 配列を優先して受け付ける実装のため、**バリデーション・保存処理（`app/record/actions.ts`）は原則変更不要**
- DBの `workout_sets` テーブル（`set_number`, `weight_kg`, `reps`）は既存のままで対応可能。**マイグレーション不要**
- 有酸素種目は `sets: []`、`distance_km: null`、`calories: null` を送る（`duration_minutes` のみ有効値）
- サブカテゴリ・器具は `muscle_sub_group_ids: []`、`equipment_id: null` を送る

## 4. 履歴サジェストの仕様

- 取得元: ログインユーザー自身の過去 `workout_exercises`（+ 関連する `workout_sets`）
- 取得条件: **頻度順 or 直近使用順**で最大**20件**（重複する種目名は1件に集約）
- 各候補が保持する情報:
  - 種目名
  - 部位（`muscle_group_id`、筋トレ時のみ）
  - 前回使用時のセット内容（重量・回数の配列。セットごとに重量が異なっていてもそのまま保持）
- 取得タイミング: `app/record/page.tsx` でサーバーコンポーネントとして事前取得し、`record-form.tsx` にpropsとして渡す
- 候補選択後、ユーザーはセット行リストを自由に編集できる（サジェストはあくまで初期値の提案）

## 5. 廃止項目一覧

| 項目 | 状態 |
|------|------|
| サブカテゴリ選択UI | 廃止（DBマスタ・列は温存） |
| 器具選択UI | 廃止（DBマスタ・列は温存） |
| 有酸素の距離入力 | 廃止（DB列は温存、保存時null） |
| 有酸素のカロリー入力 | 廃止（DB列は温存、保存時null） |
| AIメニュー提案機能 | 廃止（[SPECIFICATION_AI_SUGGESTION.md](./SPECIFICATION_AI_SUGGESTION.md) 参照。課金基盤は温存） |

**併用継続**: テンプレート機能（`app/templates/`）は本リニューアル後も引き続き利用できる。テンプレートから読み込んだ内容はper-set形式に変換してセット行リストへ反映する。
